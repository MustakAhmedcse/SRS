Section 1: Introduction to SeaweedFS and Its Role

1.1 What is SeaweedFS?

SeaweedFS is a simple yet highly efficient distributed file storage system designed specifically for storing and managing billions of small files. Unlike traditional databases that treat files as Binary Large Objects (BLOBs), which causes significant performance and scalability issues, SeaweedFS is purpose-built for fast file serving at scale with minimal operational overhead.

1.2 Key Features and Advantages

  • High Performance: Serves millions of files per day with minimal latency
  • Scalability: Easily add new storage nodes without complex migration procedures
  • Automatic Replication: Built-in data redundancy with configurable topology
  • Low Cost: Uses commodity hardware and requires minimal maintenance
  • Direct HTTP API: Simple REST API for easy integration with any application
  • S3 Compatibility: AWS SDK support enables familiar development patterns
  • Automatic Cleanup: TTL (Time To Live) support for temporary file management

1.3 Why SeaweedFS Instead of Traditional Databases?

Aspect Traditional Database SeaweedFS
Speed BLOB storage causes query bottlenecks Direct HTTP file serving, optimized for throughput
Scaling Requires expensive hardware upgrades Add commodity servers horizontally
Replication Complex manual replication setup Automatic, built-in replication
Cost High operational and licensing costs Low cost, open-source, minimal overhead
Operations Complex backup and recovery procedures Simple file-level operations

1.4 Suitable Use Cases

SeaweedFS is ideal for the following scenarios:

  • Image storage for user profiles, posts, and media galleries
  • Document management systems with file versioning
  • Video streaming platforms and video on demand services
  • Cloud backup and archival storage
  • CDN origin servers for content distribution
  • Temporary file storage with automatic expiration
  • Multi-tenant application file isolation

Section 2: Core Architecture and System Components

2.1 The Four Core Services

Master Server - The Cluster Orchestrator
The Master Server is the central coordinator for your entire SeaweedFS cluster. It maintains the overall cluster state, assigns storage locations for new files, monitors server health, and orchestrates replication. The Master Server is single-threaded for simplicity and reliability.

Master Server Responsibilities:

  • Assign unique File IDs and Volume IDs when files are uploaded
  • Determine which Volume Server will store each file based on replication policy
  • Monitor health of all Volume Servers through heartbeat checks
  • Manage automatic data replication across configured topology
  • Handle load balancing to prevent volume overflow
  • Coordinate failover when volumes become unavailable
  • Maintain cluster-wide configuration and metadata

Ports: HTTP=9333, gRPC=19333 (Internal)

Volume Servers - The Storage Nodes
Volume Servers are stateless storage nodes that hold actual file data. They are intentionally kept simple and stateless, which means they can be added, removed, or replaced without complex data migration. Each Volume Server can manage multiple volumes.

Volume Server Responsibilities:

  • Store file data on local disk
  • Serve files via HTTP GET requests
  • Accept uploads via HTTP POST operations
  • Send periodic heartbeats to Master Server
  • Replicate data to other volumes based on Master instructions
  • Delete expired files based on TTL policies
  • Report disk usage and capacity to Master

Filer Server - The File System Interface
The Filer service provides a user-friendly file system interface. Instead of dealing with raw file IDs, applications interact with familiar directory paths and file metadata. Filer translates these human-readable paths to internal file IDs.

Filer Server Responsibilities:

  • Maintain directory structure and file organization
  • Store and manage file metadata (creation time, size, owner)
  • Manage collections and bucket organization
  • Provide REST API for file operations
  • Support TTL-based file expiration
  • Implement access control and permissions
  • Integrate with Master and Volume Servers

S3 Service - AWS Compatibility Layer
The S3 service adds AWS S3 API compatibility to SeaweedFS. This allows existing applications built with AWS SDKs to work seamlessly without modification, reducing integration complexity and enabling code reuse.

2.2 How Services Communicate

Request flow in SeaweedFS:

FILE UPLOAD:
Client → Filer (POST /upload)
    ↓
Filer → Master (Which volume?)
    ↓
Master → Selects volumes based on replication policy
    ↓
Filer → Volume1, Volume2 (Write file)
    ↓
Volumes → Master (Acknowledged)
    ↓
Master → Filer (Return File ID)
    ↓
Filer → Client (File ID and metadata)

FILE DOWNLOAD:
Client → Filer (GET /file-id)
    ↓
Filer → Master (Where is this file?)
    ↓
Master → Returns volume location
    ↓
Filer → Volume (Fetch file)
    ↓
Volume → Filer → Client (Stream file)

3.2 Why Two Servers?

  • Geographic Distribution: Data survives single datacenter failures

  • Independent Scaling: Secondary can grow independently
  • Single Master: Avoids split-brain issues with multiple masters

Section 4: Understanding Replication Topology

4.1 Replication Strategy Format: XYZ

SeaweedFS uses a 3-digit replication code:

Digit Level Meaning
X Datacenter Extra copies in other datacenters
Y Rack Extra copies in different racks (same DC)
Z Server Extra copies on different servers (same rack)

4.2 Your Configuration: 010

0 = No extra datacenter copies
1 = One extra copy in different rack (within same DC)
0 = No extra server copies
This means: Each file gets 2 copies - one in rack1 and one in rack2 within dc1.

4.3 How Replication Works

WHEN YOU UPLOAD "document.pdf" (2.5MB):
1. Filer asks Master: "Where should I store this?"
2. Master evaluates with 010 policy:
   - Available: Volume1 (dc1, rack1), Volume2 (dc1, rack2)
   - Chooses: Primary → Volume1, Replica → Volume2
3. Filer writes:
   - 2.5MB → Volume1 (rack1)
   - 2.5MB → Volume2 (rack2)
4. If Volume1 fails:
   - Data survives on Volume2
   - No data loss
   - New uploads skip Volume1
This ensures file survival even if one volume fails.

4.4 Alternative Strategies

Strategy Use Case Note
000 Development/Testing only Single copy - high risk
001 Very high availability Same server, high cost
010 Recommended (our setup) Balanced redundancy
110 Multi-datacenter Extra protection
200 Disaster recovery Expensive

var builder = WebApplicationBuilder.CreateBuilder(args);

// Configure S3 Client
var seaweedConfig = builder.Configuration.GetSection("SeaweedFS");
var s3Client = new AmazonS3Client(
    new BasicAWSCredentials(
        seaweedConfig["S3AccessKey"],
        seaweedConfig["S3SecretKey"]
    ),
    new AmazonS3Config
    {
        ServiceURL = seaweedConfig["S3Endpoint"],
        ForcePathStyle = true
    });
builder.Services.AddSingleton<IAmazonS3>(s3Client);

// Configure Database
builder.Services.AddDbContext<ApplicationDbContext>(options =>
    options.UseNpgsql(builder.Configuration.GetConnectionString("DefaultConnection")));

builder.Services.AddControllers();
builder.Build().Run();

5.4 File Upload Controller

[HttpPost("upload")]
public async Task<IActionResult> Upload(IFormFile file)
{
    var filerUrl = _config["SeaweedFS:FilerUrl"];
    using var httpClient = new HttpClient();
    using var content = new MultipartFormDataContent();
    using var fileStream = file.OpenReadStream();
    
    content.Add(new StreamContent(fileStream), "file", file.FileName);
    
    var response = await httpClient.PostAsync($"{filerUrl}/upload", content);
    var responseContent = await response.Content.ReadAsStringAsync();
    var uploadResult = JsonConvert.DeserializeObject<dynamic>(responseContent);
    
    var fileMetadata = new FileMetadata
    {
        FileName = file.FileName,
        FileSize = file.Length,
        FileType = file.ContentType,
        FID = uploadResult.fid,
        UploadedAt = DateTime.UtcNow
    };
    
    _context.FileMetadatas.Add(fileMetadata);
    await _context.SaveChangesAsync();
    
    return Ok(new { fileId = uploadResult.fid });
}

5.5 File Download Controller

[HttpGet("download/{fileId}")]
public async Task<IActionResult> Download(string fileId)
{
    var fileMetadata = await _context.FileMetadatas
        .FirstOrDefaultAsync(f => f.FID == fileId);
        
    if (fileMetadata == null)
        return NotFound();
        
    var filerUrl = _config["SeaweedFS:FilerUrl"];
    using var httpClient = new HttpClient();
    var response = await httpClient.GetAsync($"{filerUrl}/{fileId}");
    var fileContent = await response.Content.ReadAsByteArrayAsync();
    
    return File(fileContent, fileMetadata.FileType, fileMetadata.FileName);
}

Section 6: File Upload/Download Workflow and Processes

6.1 Upload Process

  1. User selects file in ASP.NET application
  2. Application validates file (size, type, extension)
  3. Application calls Filer: POST http://27.147.159.197:8888/upload
  4. Filer receives and asks Master: "Where to store?"
  5. Master checks replication (010) and selects volumes
  6. Filer writes to Volume1 (rack1) and Volume2 (rack2)
  7. Volumes acknowledge successfully
  8. Master records file metadata
  9. Filer returns File ID to application
  10. Application saves metadata to database
  11. Application confirms to user: "Upload complete"

6.2 Download Process

  1. User requests file with File ID
  2. Application queries database to find file
  3. Application calls Filer: GET http://27.147.159.197:8888/{fileId}
  4. Filer asks Master: "Where is this file?"
  5. Master returns: Volume1 or Volume2 location
  6. Filer fetches from primary Volume1 (or Volume2 if primary down)
  7. Volume streams file to Filer
  8. Filer streams to application
  9. Application streams to user browser
  10. Browser downloads file with original filename

6.3 Data Reliability

With 010 replication:

  • Single volume failure: Data survives on replica in different rack
  • Entire rack failure: Data lost if using 010 (no other backup)
  • Add 3rd volume in dc2: Increases protection (consider upgrading to 100 or 110)
  • Backup strategy: Regular backups to external storage recommended

Section 7: Advanced Configuration and Customization

7.1 Volume Size Configuration

Default volume size is 30GB. To change:

Add to master command in docker-compose.yml:

-volumeSizeLimitMB=102400    # 100GB per volume
-volumeSizeLimitMB=51200     # 50GB per volume

7.2 TTL (Time To Live)

Automatically delete files after specified time:

POST /upload?ttl=3h   # Delete after 3 hours
POST /upload?ttl=7d   # Delete after 7 days
POST /upload?ttl=30m  # Delete after 30 minutes

Useful for: Temporary uploads, caching, GDPR compliance

7.3 Collections/Buckets

Organize files into collections:

POST /dir/make?collection=images
POST /upload?collection=images
GET /images/        # List collection

Section 9: Performance Optimization and Scaling

9.1 Performance Tuning

  • Monitor disk I/O - use iostat to check volume performance
  • Monitor network bandwidth - ensure sufficient capacity between servers
  • Implement connection pooling in ASP.NET HTTP clients
  • Use compression for large files when appropriate
  • Batch small file uploads together
  • Implement caching layer with CDN for frequently accessed files
  • Monitor Master Server CPU usage - should remain low

9.2 Scaling Strategies

  • Vertical: Increase volume size limits or hardware resources
  • Horizontal: Add new volume servers with proper datacenter/rack tagging
  • Multi-region: Add entirely new datacenters for geographic distribution
  • Load balancing: Run multiple Filers behind a load balancer
  • Monitoring: Set up Prometheus/Grafana for metrics collection

9.3 When to Add Storage

Add new volumes when:

  • Total usage reaches 80% of capacity
  • Upload operations become slow
  • Master Server begins showing disk pressure

Section 10: Best Practices and Frequently Asked Questions

10.1 Best Practices

  • Always use replication - Never use 000 in production
  • Monitor actively - Set alerts for disk usage, errors, response times
  • Plan capacity - Upgrade before reaching critical limits
  • Regular backups - Test restore procedures quarterly
  • Load balance Filers - Avoid single point of failure
  • Cleanup old files - Use TTL for temporary data
  • Version control configs - Track docker-compose.yml changes
  • Secure credentials - Use secrets manager for S3 keys
  • Test failover - Practice disaster recovery procedures

10.2 Backup and Disaster Recovery

Backup Frequency: Daily for volumes, weekly for full backup
What to backup: Volume data (/data1, /data2), database metadata
Where: External storage (S3, cloud, external disk)
Recovery: Restore volumes, restart services, resync metadata

10.3 Security Considerations

Network: Keep SeaweedFS on internal network, use reverse proxy externally
HTTPS: Use nginx with SSL/TLS in production
Credentials: Store S3 keys in secrets manager, never in code
Firewall: Allow only necessary ports, restrict by source IP
Authentication: Implement at application layer, not SeaweedFS
Encryption: Use HTTPS for transport security

10.4 FAQ

How long are files stored?
Permanently until deleted or TTL expires

Can I use multiple Filers?
Yes, put behind load balancer

Max file size?
Limited only by disk space, no hard limit

Is S3 fully compatible?
Most features work, some advanced features not supported

Can I mix volumes of different sizes?
Not recommended, keep consistent

What if Master fails?
Cluster read-only, optional failover with backup master

Does SeaweedFS encrypt data?
No, use HTTPS for transport, disk encryption for at-rest

How do I migrate data?
Copy files to new volume, or use S3 tools for bulk operations
