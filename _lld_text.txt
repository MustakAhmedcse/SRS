### [Heading 1] 1. Introduction

### [Heading 2] 1.1 Purpose
This Low-Level Design (LLD) explains, in detail, how the Sales Commission Automation Platform (SalesCom) is built. For every feature it describes the screens, the database tables, how the feature works step by step, the APIs it uses, and the rules it must follow. Developers use this document as the guide for building the system.

### [Heading 2] 1.2 Scope
SalesCom lets business users build their own commission reports through a step-by-step wizard, replacing the old manual SQL process. This document covers seven features - Login, Data Sources, Reports, Approval, Dashboard, Notification and Disbursement - plus shared concerns and the overall code structure.

### [Heading 1] 2. System Architecture

### [Heading 2] 2.1 Architecture Overview
SalesCom is built in layers. A web app (React/Next.js) talks to a C#/.NET backend over a secure HTTPS connection. The backend keeps all report settings and run history in a PostgreSQL database, keeps uploaded files and result files in file storage (SeaweedFS), and runs background jobs with Hangfire. The services talk to each other through a message queue (RabbitMQ). A small Python service turns each report's saved settings into SQL, which then runs on the database.
Web app (Next.js) <— HTTPS —> .NET backend <— Dapper/EF Core —> PostgreSQL database. The backend also uses file storage (SeaweedFS) and a message queue (RabbitMQ); background jobs (Hangfire) handle runs, the SQL service and notifications (SMS/Email).

### [Heading 2] 2.2 Technology Stack
[TABLE]
| Layer | Technology | Notes |

| Data access | Dapper + Entity Framework Core | Dapper for read queries; EF Core for ORM and migrations |

### [Heading 2] 2.3 Common Conventions
Every API address starts with /api/v1 and returns data in JSON format.
Unless stated otherwise, every record uses a UUID as its unique id.
All dates and times are stored in UTC and written in the standard ISO 8601 format.
All money amounts are stored as numbers; the currency is always BDT.
Every request includes the login token (JWT); the backend works out who the user is and what they can do.
Nothing is permanently deleted - records are marked inactive instead, so history is never lost.

### [Heading 1] 4. Authentication and Session

### [Heading 2] 4.1 Overview
SalesCom is internal-only, so everyone signs in through the company's Central Login using a one-time password (OTP). The web app collects the username and password and sends them to the backend; the backend (which holds the secret application name and key) talks to Central Login and, once the OTP is verified, issues its own SalesCom login token (JWT) for the web app to use on every request. The full step-by-step flow is in 4.4.

### [Heading 2] 4.2 User Interface
Login screen (web app): Username, Password (with a show/hide option) and a Remember-me box. The form is sent to the SalesCom backend, never straight to Central Login.
The OTP step belongs to Central Login: the user's browser is sent to the Central Login page, which shows the OTP box, the countdown and the resend option. SalesCom has no OTP screen of its own.
After signing in, the user lands on the Dashboard, which shows when they last signed in successfully and when a sign-in was last refused.

### [Heading 2] 4.3 Data Model
User identity and access rights come from Central Login and are kept up to date in the local user and user_right tables by the hourly sync service (see 4.4.4). At sign-in the looks the user up here. Tables:

### [Heading 2] 4.4 Process Logic

### [Heading 3] 4.4.1 Login and SSO Redirect
The web app sends the username, password and remember-me choice to the backend at POST /api/v1/auth/login.
The backend adds the secret application name and key and calls Central Login at POST /account/v1/login.
Since all users are internal, Central Login responds with a redirect link (no token yet). The backend passes this link to the web app, which sends the user's browser to the Central Login OTP page.

### [Heading 3] 4.4.2 OTP Verification and Callback
Central Login handles the OTP on its own page (entering the code, the countdown and resend). Some non-secret details may be added to the return link.
After a correct OTP, Central Login sends the browser back to SalesCom at GET /api/v1/auth/callback with a one-time authToken.
The backend calls Central Login at POST /account/v1/verify-auth-token with that token to confirm the login and get the user's profile.
Any login tokens Central Login returns here are ignored - SalesCom only needs the user's profile.

### [Heading 3] 4.4.3 SalesCom JWT Issuance
Once the token is confirmed, the backend looks up the user and their access rights (kept current by the hourly sync) and creates its own SalesCom login token (JWT) expire time of one day(24 hour) holding the user id, username and an expiry time, signed with a backend secret.
The backend gives this token to the web app as the result of login; no Central Login token is ever shown to the browser.
The web app saves the SalesCom token and sends it with every request. The backend checks the token on each request and loads the user and their rights.
For inactivity in frontend for 4 hours, user will be logout means backend token delete from frontend.
When the token expires/logout, the user simply signs in again from the start.
Every sign-in attempt - successful or refused - is written to the login table (user, time, status, remarks) for the Dashboard's Login Attempts card.

### [Heading 3] 4.4.4 User Provisioning
A background service runs every hour to keep each user's details and access rights up to date with Central Login.

### [Heading 2] 4.5 API Endpoints
There are two sets of endpoints: Central Login's own service (called only by the backend) and SalesCom's backend endpoints that the web app uses.
[TABLE]
| Method | Path | Purpose |

### [Heading 2] 6.4 Process Logic

### [Heading 3] 6.4.1 Step 1 - Basic Input
Enter the report name (must be unique), commission cycle, channel, and start and end dates (start must not be after end).
Recurrent switch: when on, choose how often it repeats (daily/weekly/monthly). EV Disbursement switch: when on, choose a disbursement time. Also input SMS content when EV disbursement is on.
User can only active one disbursement EV/POS at a time.
'Save as Draft' saves the report as a draft and gives back the report id used through the rest of the wizard.

### [Heading 3] 6.4.2 Step 2 - Supporting Upload
Drag and drop or browse for a CSV file; the first row must be the column names. You can upload several files (shown with serial number, original and source file names, and a Remove option before saving).
Limits: up to 30 columns and 500 MB per file.
A preview shows the first rows, with each column's type detected automatically (text/number/date). The user can change a column's type, and any value that does not match its type is flagged as a warning.
When confirmed, the original file is kept in file storage and a database table is created from its rows, so it can be used as a data source in later steps.

### [Heading 3] 6.4.3 Step 3 - Achievement Calculation
Build one or more Achievement blocks (ACH#1, ACH#2, …); for each, pick a data source (a registered source or a CSV from step 2).
Each block has pipeline steps shown as expandable cards: Filter, Combine Data, Summarize, Calculate, Modify.
Filter: Column, Condition/operator (Is one of, Equals, Not equals, Greater than, Less than, ≥, ≤, Between), and the Compare-to value(s).
Combine Data: Get Data From (another registered source, an uploaded CSV, or an existing Achievement); the join condition as Current column = other column. How to Combine — Keep only matched rows (inner join) or Keep all rows (left join).
Summarize: Get Data From; a Result Column name for the output; a Calculation (Count Rows, Sum, Avg, Min, Max); an optional Group/Form Column to group by.
Calculate: All calculation can be done here Math Formula, IF / CASE, rounding etc.
Modify : modify column value.
Each block outputs panel shows the columns the block will produce (name, type). An existing Achievement can be used as input to another.
Blocks can only be removed from the end.
Before moving on, the system checks every pipeline to make sure the columns and fields it refers to actually exist.

### [Heading 3] 6.4.4 Step 4 - Incentive Calculation
Build one or more Incentive blocks using the same steps: Filter, Combine Data, Summarize, Calculate, Modify as before.
Incentive blocks read one or more Achievement blocks (or a system source or supporting upload); an existing Incentive can feed another.
The Final block maps any produced column onto the final commission table (channel code and commission amount), grouped per channel.

### [Heading 3] 6.4.5 Step 5 – Preview and Schedule
After Submit, a confirmation screen offers Back to Reports, Schedule for Later.

### [Heading 3] 6.4.6 Run Config
Run Now is only available once the report is fully approved. Only one run happens at a time; extra runs wait in a priority queue. A Demo Run is just a test - it does not disburse.
The Schedule window takes a date and time. Recurrent reports run repeatedly between a start and end date on the chosen frequency; non-recurrent reports run once on schedule single date.
Only with access right user can edit a schedule or stop report; every change is recorded, along with who started each run.

### [Heading 3] 6.4.6 Run Execution Lifecycle
This is the full path a report run takes, from start to payout. It connects the setup, stage, run, result and disburse tables.
Step 1 - Trigger: Run Now (after full approval) or scheduler; type = Final or Demo; triggered_by = UN or System.
Step 2 - Queue: one run at a time; others wait in a priority queue.
Step 3 - Create run: insert report_run (run_status = pending → running).
Step 4 - Snapshot stages: copy every stages row into run_stage (run_status = not run), freezing the SQL.
Step 5 - Execute in order: run each run_stage SQL sequentially; each stage materialises a temp table (output_table_name); status not run → running → succeeded/failed.
Step 6 - Export outputs: each succeeded stage is written as a CSV to object storage (bucket, object_url, file_name, file_generated_at).
Step 7 - Final commission: the final stage writes one row per channel into final_commission (channel_code, commission_amount).
Step 8 - Cleanup: drop temp tables (cleanup_status); set report_run = completed (or failed with error).
Step 9 - Disbursement (Final + approved): EV writes ev_disburse rows and SMSes recipients; POS builds a CSV from final_commission and records pos_disbursement.

### [Heading 3] 6.4.7 Report Detail View and Cloning
Details page tabs: Basic, Supporting Uploads, Achievements, Incentives, Run Log, and a single Disbursement tab (EV or POS) shown only when one is turned on.
The Run Log lists each run (Final or Demo), newest first, with download options per file and a Download All.
Clone copies the whole setup (blocks, field and data-source references, approval levels) into a new draft with a new unique name. Uploaded files, past runs, approvals and payouts are not copied.

### [Heading 2] 7.4 Process Logic
Setup and start:
A report is given an Approval Flow when it is created. When the report is send for Approval, the system opens one approval request for that report, linked to its flow, sets the current level to the flow's first level (lowest order), records who submitted it and when, and sets the overall status to "in progress" (waiting for the first level).
Each level has an approval type, and every type has a phase: PRE_RUN (approve the setup before the report can run) or POST_RUN (approve the results after a run, before payout). The current level's type decides which phase the request is in. There is only one request per report; it walks all the levels in order.
Who can see and act:
A request is shown only to the users assigned to its current open level (Approval Level User); no one else sees it.
Approval is sequential: it starts at the first level and moves up one level at a time. A higher level cannot act until every lower level has approved.
Approve:
When an assigned approver approves, the system records the decision (level, who, comment if any, time) and moves the current level to the next one — so the request now appears for that next level's users.
When the final level of the whole flow approves, the request's overall status becomes approved and an "approval completed" event is published.
Reject / Request changes:
An approver must enter a comment to reject.
On reject, the request goes back to the previous level (or back to the draft if it is rejected at the first level), the decision and comment are recorded, the overall status becomes Rejected / Changes Requested, and the Maker is notified by email.
The Maker fixes the issue and resubmits, which restarts approval from the first level.
Report Logic:
PRE_RUN levels are approved once, before the report runs; after they all approve, the report can run on demand or on schedule.
For a non-recurrent report whose flow also has POST_RUN levels: after a Final run produces results, the same request continues into the POST_RUN levels; only after they all approve does disbursement run.
A recurrent report must use a flow with PRE_RUN levels only. Its setup is approved once; after that, every scheduled run executes and pays out automatically, with no result approval.
A Demo run never needs approval and never Disburse.
Every decision is stored (approver, comment, time) so the full approval history can be shown.
Email is sent when approval is requested, when it is rejected, and when it is completed.

### [Heading 1] 11. Asynchronous Services and Events

### [Heading 2] 11.1 Overview
Heavy or time-based work is kept out of the main request path by using a message queue (RabbitMQ) and timed jobs (Hangfire). The API just sends a message and saves status; separate services do the actual work. This keeps the app fast and lets runs be handled by priority.

### [Heading 2] 11.2 Event and Queue Topology
[TABLE]
| Event | Publisher | Queue (priority) | Consumer |
| report.saved | Reports API (on save) | q.sql-generate | SQL Generator |
| run.requested | Runs API (Run Now) | q.run.high | SQL Executor |
| run.requested | Runs API (Demo Run) | q.run.mid | SQL Executor |
| run.requested | Scheduler (Hangfire) | q.run.low | SQL Executor |
| ev.disburse | EV timer (Hangfire) | q.ev-disburse | EV Disburse Service |
[/TABLE]

### [Heading 2] 11.3 SQL Generator Service (Python)
When a report is saved, the API sends a 'report saved' message with the report id onto the queue.
The SQL Generator picks up that message, turns the report's settings into SQL, and saves it step by step into the stages table.

### [Heading 2] 11.4 SQL Executor Service
When a Demo Run, Run Now or scheduled run starts, the API sends a 'run requested' message to a priority queue (Run Now = high, Demo = medium, schedule = low).
The Executor handles these in priority order, running the report through the Run Execution Lifecycle (Section 6.4.6).

### [Heading 2] 11.5 EV Disburse Service
A timed job checks whether any approved Final run has EV payout turned on and its payout time has arrived; if so, it sends an 'EV payout' message.
The EV Disburse service picks this up and pays out from the run's final commission, recording each payout and sending the SMS (Section 10).

### [Heading 2] 11.6 POS Disburse Job
A timed job checks whether any approved Final run has POS payout turned on; if so, it reads the run's final commission, builds a CSV and saves it to the set location.
The job records the result (file location, status and time).

### [Heading 2] 11.7 User Sync Service
A job runs every hour to copy user details and access rights from Central Login into the local tables, so changes take effect within an hour.

### [Heading 1] 14. Appendix

### [Heading 2] 14.1 Glossary
Achievement block: a set of steps that works out a performance figure.
Incentive block: a set of steps that works out a payout from achievements, usually using slab/case rules.
Final commission: the per-channel amount produced by the last step of a run.
EV / POS disbursement: the two payout channels (Electronic Value / Point-of-Sale).
